除了 ERP，这段时间另一个重点是工业机器视觉系统的开发与现场交付。视觉项目最大的反差是——**实验室里能跑 10 万次的程序，到产线上 1 小时就崩**。

这篇把我踩到的 5 个稳定性陷阱写下来。

## 陷阱 1：相机曝光时间被环境光"偷走"

实验室是恒定 LED 顶灯，曝光参数调好就好。到了车间——头顶有窗户、有顶吊起重机、有不规律开关的车间灯——**实测白班和夜班的环境照度能差 3 倍**。

原本写死的 `exposure_time = 8ms` 在白班直接过曝，黑漆漆一片。

**修法**：

- **加遮光罩**。物理隔绝优先于软件补偿，简单粗暴有效。
- **自动曝光不要全靠相机驱动**。我们写了一个小循环：每 N 张图取灰度均值，超出阈值就微调曝光，平滑过渡避免抖动。
- **暗箱比明箱稳**。能让客户接受全封闭工位，就别留个小窗。

## 陷阱 2：上位机 / 下位机的串口包"粘在一起"

PLC 与上位机走 RS485/Modbus，理论上协议清晰。实战中两个坑：

1. **TCP-Serial 转换器**默认有 50ms 缓存。短报文在它内部攒一波再吐，造成上位机看到的是"两条粘在一起的数据"。
2. **应用层没做帧校验**。结果就是偶发性"突然解析失败"，调试到怀疑人生。

**修法**：

```python
# 不要相信 read() 一次返回的就是完整一帧
# 应用层加帧头 + 长度 + CRC + 帧尾
def parse_frame(buf):
    while b'\xAA\x55' in buf:
        idx = buf.index(b'\xAA\x55')
        if len(buf) - idx < 6:  # 帧头 + 长度 + 最小 payload
            break
        length = buf[idx+2]
        if len(buf) - idx < length + 4:
            break  # 不完整,等下次
        frame = buf[idx:idx+length+4]
        if crc_check(frame):
            yield frame
        buf = buf[idx+length+4:]
    return buf
```

关键：**`recv()` 返回的字节流不是消息单元**。应用层必须做帧拼接。

## 陷阱 3："看起来没问题但就是慢"——内存泄漏

视觉算法吃图，OpenCV 的 numpy 数组如果在多线程里乱传引用，**计数器对不上 GC 不会回收**。

症状：开机半小时 OK，开机 8 小时画面卡顿，第二天早班 OOM 重启。

**修法**：

- 每个 frame 处理完显式 `del frame; gc.collect()`（开销可以接受，比 OOM 划算）
- 视频缓冲区**限定大小**，用 `collections.deque(maxlen=30)`
- 加 prometheus 或自写监控，**进程内存超阈值自动重启**（systemd 或 supervisord 配置）

> 教训：宁可每天定时重启一次，也不要追求"7×24 不重启"。生产稳定的关键是**可恢复**，不是不出错。

## 陷阱 4：日志不带时间戳和工件 ID = 灾难

第一版日志：

```
[INFO] inspection passed
[INFO] inspection failed
[ERROR] camera timeout
```

现场出问题后让客户翻日志——根本找不到对应哪一件产品。

**修法**：每条日志强制带 `工件 ID + 工序 + 时间戳（精确到 ms）+ 算法版本号`。

```python
logger.info("INSPECTION", extra={
    "part_id": part_id,
    "process": "step_3_diameter",
    "result": "PASS",
    "score": 0.987,
    "algo_ver": ALGO_VERSION,
})
```

JSON 格式落盘，方便 grep / 倒灌 ES 做后期分析。**当下多花的 10 分钟，省了未来 100 倍的扯皮。**

## 陷阱 5：现场支持的 "5W 不要"

最后一条是关于人的不是技术的：

- **不要**远程指导客户改代码。差一个符号现场就停产。
- **不要**没备份直接改配置。一定先 `cp config.yaml config.yaml.bak.$(date +%s)`。
- **不要**凭客户描述判断问题。一定要看日志/截图/复现步骤。
- **不要**在车间直播调试新算法。先在测试机过一夜再上。
- **不要**修完不写复盘。每次现场问题都要回头沉淀，否则下次还会踩。

## 收束

视觉项目的难，不在算法本身——OpenCV、深度学习模型都是现成的。难的是**让这一套东西在产线上 24 小时跑下去不出岔子**。这部分经验只能靠现场跑出来。
