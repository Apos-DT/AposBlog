来青岛火一五的这段时间，我主要在做制造业客户的 Odoo ERP 项目。从需求调研、模块二开到上线交付，把销售-采购-库存-生产-质检-财务这条主链路跑通。这篇把过程中的关键决策和踩坑写下来，给同样在做 Odoo 二开的同行参考。

## 一、为什么是 Odoo，而不是 SAP / 用友 / 自研

客户是中小型制造企业。最初他们用 Excel + 钉钉审批跑生产，痛点很集中：

- **数据散在十几个表格里**，订单一改要手动同步 5 处
- **现场没有实时库存**，缺料经常等到工单开始才发现
- **质检数据不沉淀**，每批次的合格率靠车间主任拍脑袋

我们评估过 SAP Business One 和用友 U8，最终选了 Odoo Community + 自定义模块：

| 维度 | Odoo 优势 |
|---|---|
| 成本 | Community 版免费，二开成本可控 |
| 灵活度 | Python + ORM，自定义字段/视图/工作流非常顺手 |
| 集成 | 自带 REST/XML-RPC，与企业微信、外部 BI 对接方便 |
| 生态 | OCA 社区模块多，能站在巨人肩上 |

代价是——**很多业务场景需要二开**，原生模块满足不了。这才是工作真正开始的地方。

## 二、需求调研：第一周做什么

新手做 ERP 项目最容易犯的错，是**直接跳到"开发什么功能"**。

我的做法是先用三天画清楚客户的真实流程：

1. **跟车间主任走一遍产线**。从领料、上料、加工、首检、批量生产、终检、入库——每个节点谁操作、用什么单据、卡在哪里。
2. **画 BPMN 流程图**。不是为了好看，是逼自己回答"这一步如果跳过会怎样"。客户经常说"这一步我们一定要做"，但你追问为什么，常常发现是历史包袱。
3. **角色权限矩阵**。生产经理能看库存成本吗？质检员能改工单状态吗？这些表格做出来后，比写 100 行 prd 都管用。

> [!NOTE] 教训:早期我跳过流程图直接开发,结果做完审批流发现客户车间根本没有"组长"这个角色,整个流程要返工。

## 三、二开重点：哪些必须做、哪些不要做

原生 Odoo 已经很强，**不要见一个需求改一个**。判断要不要二开的标准：

**做二开 ✅**：
- 字段缺失（如客户特定的 `批号 / 炉号 / 检测报告编号`）
- 审批流不匹配（如多级会签）
- 业务规则不同（如同一物料按客户报不同价）
- 报表口径不同（财务月报、产能日报）

**不做二开 ❌**：
- 仅 UI 不习惯（先培训用户）
- 已有 OCA 模块覆盖（直接装）
- "如果以后用得到"（YAGNI——你不会用到）

> [!TIP] 一个判断标准:**这个需求三个月后还会存在吗?** 如果你犹豫,大概率不会。不做。

实际做的几个关键二开：

### 1. 销售订单 → 生产工单的自动拆分

客户的成品由多个半成品组装，原生 MO（Manufacturing Order）一对一拆分对他们太粗。我加了一个`生产路线规划`字段，根据 BOM 自动拆出多级 MO 并串联依赖关系。

```python
class SaleOrder(models.Model):
    _inherit = 'sale.order'

    def action_confirm(self):
        res = super().action_confirm()
        for line in self.order_line:
            if line.product_id.routing_type == 'multi_level':
                self._create_cascading_mos(line)
        return res
```

关键不是代码，是**继承 `_inherit` 而不是重写**——这样原生升级不会冲突。

### 2. 字段扩展：批号 / 炉号 / 检测报告

每个客户都有自己的"行业方言"。新建一个模块 `mfg_batch_traceability`，给 `stock.production.lot` 加一组字段，并在收发料时强制录入。

工程口诀：**字段扩展永远新建模块，不要去改 base 模块**。

### 3. 企业微信集成

客户内部沟通在企微，要求工单异常推送到企微群、审批通过推送到当事人。

走 Odoo 的 `mail.thread` 钩子，监听记录变更事件 → 通过企微机器人 webhook 推送：

```python
@api.model
def _post_to_wecom(self, message, mentioned_userid):
    payload = {
        "msgtype": "markdown",
        "markdown": {"content": message},
        "mentioned_list": mentioned_userid,
    }
    requests.post(WECOM_WEBHOOK, json=payload, timeout=5)
```

注意点：

- **错误必须可重试**。生产推送不能因为网络抖动丢消息，加 `mail.queue` 落库重试
- **日志带 record ref**。出问题能快速定位到具体单据
- **webhook URL 走配置**，不要硬编码（测试/生产环境不同）

## 四、数据迁移：最容易被低估的工作

客户有几年的历史数据，从 Excel 迁到 Odoo。我做的几件事：

1. **写迁移脚本而不是手工导入**。用 `xlrd` 读 → 校验 → 调 `env['xxx'].create()` 落库。这样数据有问题可以全量回滚重跑。
2. **先迁主数据再迁单据**。客户/供应商/物料/BOM → 库存余额 → 历史订单。顺序反了就要被外键约束教做人。
3. **留 `legacy_id` 字段**。每条记录记原 Excel 行号，方便客户对账。

> [!WARNING] 生产环境 truncate 前必备份,备份完再 backup 一遍。我见过一个项目因为漏了第二步在迁移中途断电,折腾到凌晨四点才追回当天数据。

## 五、上线节奏：分阶段 vs 大爆炸

不要"一次性切换"。我们的节奏：

- **Week 1-2**：销售 + 采购模块上线，财务并行 Excel 校验
- **Week 3-4**：库存 + 生产模块上线，老系统只读保留
- **Week 5-6**：质检 + 财务正式切换，老系统下线

每个阶段都留**双轨期**（新老系统并行），出问题立刻回退。

## 六、写给同行的几条经验

1. **客户嘴里的"很简单"通常很复杂**。多问几个为什么，能省一半返工。
2. **不要直接 fork Odoo 源码**。改 base 等于自杀，永远继承扩展。
3. **每个模块的 `__manifest__.py` 都要写依赖**。漏写 depends 上线后炸的概率 80%。
4. **`-u module_name` 升级模块时记得加 `--stop-after-init` 跑一次**，先在沙箱验证无报错。
5. **`ir.model.access` 与 `ir.rule` 是两层**——前者管模型级别 CRUD，后者管记录级过滤。admin 一样会被 `ir.rule` 拦。
